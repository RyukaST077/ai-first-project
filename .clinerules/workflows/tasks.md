# 目的と役割

> 目的：実装計画書のタスクを **Atomic Commit 単位の作業タスクリスト** に分解し、並行開発できる形で出力する。
> 役割：シニア・エンジニア/スクラムマスター。

---

## 入力パラメータ（必須）

`TASK_ID`：`docs/implements_plan.md` に記載されているタスクID（例：`TASK-123`）

⚠️ `TASK_ID` が未提示の場合は提示を求めて終了（タスクファイルは作成しない）。

出力先：`_tasks/<TASK_ID>.md`（このファイルのみ作成/更新）

---

## 絶対ルール

1. **設計書の情報を優先**する。記載がない仕様は `TBD` で明示・推測補完禁止。
2. **TBD が残る状態では実装着手しない。** ドラフト作成は可、確定・実装は全TBD解消後。
3. **1コミット = 1目的（Atomic Commit）**。機能追加/リファクタ/テスト改善/ドキュメント整理/インフラ変更を混在させない（目的達成に必須な最小テスト/ドキュメントは同コミット可）。
4. **受け入れ基準はターミナルで自動検証可能**にする（`コマンド → exit code/出力`形式）。目視・ブラウザ・主観判断は禁止。
5. **コミットはビルド/テストが通る単位**で刻む。
6. **コミットメッセージは Conventional Commits 形式**：`<type>(<scope>): [<TASK_ID>] <description>`
   - type: `feat/fix/refactor/docs/test/chore/ci/style/perf/build`
7. **スコープを厳守（Task Fence）**：他タスクの機能・画面・APIを実装しない。YAGNI原則。迷ったら実装しない。
   - 禁止：実装計画書にない機能追加 / 他タスクの機能・画面・API実装 / 先行実装 / 他タスク所有ファイルの変更 / スコープ外のリファクタ・整形 / OUT OF SCOPEの実装
8. **出力は `_tasks/<TASK_ID>.md` のみ**。テンプレートのセクションは省略しない。

---

## 回答フォーマット（毎回必須）

1. **フェーズ**（現在番号）
2. **確認ゲート結果**
3. **スコープ境界**（IN SCOPE / OUT OF SCOPE / BOUNDARY）
4. **内容**（タスク情報 / コミット分割案 / ドラフト）
5. **参照設計書**（根拠）
6. **TBD一覧**
7. **TBD解消のための質問**（TBDがある場合）
8. **次のアクション**

---

## フェーズ0：タスク情報の確認

1. `docs/implements_plan.md` から `TASK_ID` を検索し、優先度/対象/依存関係/方式・観点を抽出
2. 関連する設計書を特定（`docs/implements_plan.md`、`docs/design/basic_design.md`、`docs/design/` 配下）

**タスクIDが存在しない場合**：`_tasks/` ファイルを作成せず「タスクID `TASK_ID` が見つかりません」を出力して終了。

**初回回答に含めること**：(a) タスク情報の要約（ID/対象/概要/依存/後続）(b) 参照設計書と抽出仕様 (c) TBD一覧 (d) スコープ境界 (e) コミット分割案（C-001〜C-020）

**確認ゲート0**

| チェック | 検証方法 | 期待結果 |
|----------|----------|----------|
| タスクIDが存在する | `grep -n "$TASK_ID" docs/implements_plan.md` | 1件以上ヒット |
| 依存関係が記載されている | 該当タスク行の「依存」列を確認 | 空欄でない or「なし」と明記 |
| 分割戦略の根拠が確認できる | 設計書の記載を確認 | `split_strategy` 選定理由が記述できる |

---

## フェーズ0.5：スコープ境界の定義

1. 依存元・依存先・同一フェーズのタスクを抽出し、境界ルールを定義
2. IN SCOPE / OUT OF SCOPE / BOUNDARY テーブルを作成
3. 共有リソースのオーナーシップを明確化

**スコープ判断フロー（各変更に適用）**
```
1. TASK_ID の受け入れ基準を満たすために必要か？ → No → 実装しない
2. 最小限必要か？ → No → 実装しない
3. 他のTASK_IDで実装予定か？ → Yes → 実装しない
4. 上記すべてクリア → 実装する
```

**確認ゲート0.5**

| チェック | 期待結果 |
|----------|----------|
| IN SCOPE定義済み | 1件以上の項目が記載 |
| OUT OF SCOPE定義済み | 1件以上（担当タスクID付き） |
| BOUNDARY定義済み | 各リソースに権限明記 |
| 隣接タスクとの重複なし | 重複する機能ID/画面ID/API IDがない |

⚠️ **スコープ境界が未定義の場合、フェーズ1に進まない**

---

## フェーズ1：設計情報の収集

1. `docs/` 配下の関連設計書を読み込み、仕様・制約・I/F・データ構造を抽出
2. 不足情報を `TBD` として記録

**設計情報が不足している場合**：`TBD` として記録 → 関連設計書を追加検索 → それでも不足ならユーザーに質問して解消

**確認ゲート1**

| チェック | 検証方法 | 期待結果 |
|----------|----------|----------|
| 関連設計書が存在する | `ls docs/` で確認 | 必要ファイルが存在 |
| 設計書に該当IDが記載 | `grep "<ID>" <設計書パス>` | 1件以上ヒット |
| 不足情報をTBD記録済み | タスクファイルを確認 | TBDセクションに記載 |

---

## フェーズ1.5：TBD解消

1. TBD一覧をユーザーに提示し質問して解消する
2. 解消内容を `docs/` 配下の設計書に反映する
3. 全TBD解消後にフェーズ2へ進む

**確認ゲート1.5**

| チェック | 検証方法 | 期待結果 |
|----------|----------|----------|
| 全TBD解消済み | TBDセクションを確認 | 空 or 全「解消済み」 |
| 設計書に反映済み | `git diff docs/` | 解消内容が追記されている |

⚠️ **TBDが残っている場合、フェーズ2に進まない**

**TBD質問フォーマット**：
📋 **TBD解消のための質問**
- **TBD-001**: <具体的な質問>（選択肢: <参考情報>）

---

## フェーズ2：コミット分割の設計

1. **`task_type` / `split_strategy` の確定**（計画書・設計書のキーワードから1つ選定し理由を記載）
   > `task_type`：タスク全体の性質 / `split_strategy`の enum: `new_feature/change/bugfix/refactor/infra/migration/quality/security/release/other`
2. **「土台 → 中核 → 結合 → 仕上げ」** 順でコミット設計（最大20個、超える場合は統合）
3. 直列必須 / 並行作業可能なコミットを識別

**受け入れ基準の記述ルール**：ターミナルで実行可能なコマンドのみ（目視・ブラウザ確認・主観判断は禁止）

| 種別 | コマンド例 | 期待結果 |
|------|-----------|----------|
| ビルド/テスト | `npm run build` / `npm test` | exit code 0 |
| 型チェック/Lint | `npx tsc --noEmit` / `npm run lint` | exit code 0 |
| ファイル生成確認 | `test -f <path> && echo ok` | `ok` |
| API応答確認 | `curl -s -o /dev/null -w "%{http_code}" <url>` | `200` |
| DB状態確認 | `npx prisma migrate status` | `No pending migrations` |

**確認ゲート2**

| チェック | 検証方法 | 期待結果 |
|----------|----------|----------|
| split_strategy確定 | タスクファイルを確認 | enumのいずれか |
| 1コミット = 1目的 | 各コミットの「目的」を確認 | 単一目的のみ |
| コミット数 ≤ 20 | `grep -c "^## C-" <ファイル>` | 20以下 |
| 受け入れ基準が自動検証可能 | 「検証コマンド」列を確認 | ターミナルコマンドのみ |
| 全コミットがIN SCOPE内 | スコープ境界と照合 | OUT OF SCOPEに触れるコミットがない |
| BOUNDARY権限に違反なし | BoundaryテーブルとDiffを照合 | 「参照のみ」ファイルを変更しない |

---

## フェーズ3：タスクファイルの作成

テンプレートに沿って `_tasks/<TASK_ID>.md` を作成。受け入れ基準の「検証コマンド」列はターミナルコマンドのみ記載。

**確認ゲート3**

| チェック | 期待結果 |
|----------|----------|
| 全セクションが埋まっている | TBD・空欄なし |
| 受け入れ基準が自動検証可能 | ターミナルコマンドのみ、目視確認なし |
| スコープ境界セクションが完備 | IN/OUT/BOUNDARYが全埋まり |

---

## 制限

| 項目 | ルール |
|------|--------|
| 仕様の補完 | 設計・計画にない内容を推測で追加しない（不明点は `TBD` + チェックリスト化） |
| コミット数 | 最大 `C-020` まで（超える場合は統合して調整） |
| 出力 | `_tasks/<タスクID>.md` のみ（他ファイルは作成しない） |
| テンプレ | セクションは省略しない |
| **スコープ** | **IN SCOPEに含まれない変更は一切行わない。他タスクの実装範囲に属する機能・ファイルは触れない** |
| ファイルパス | 変更予定ファイルは必ずパスで書く（不明なら `TBD`） |
| 更新日/バージョン | 実行当日の日付（YYYYMMDD）、`1.0.0` から開始 |
