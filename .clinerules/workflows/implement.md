## 0. 前提（あなた=AIが守るべきルール）

1. **コミットタスクは番号順（C-001〜）に1つずつ処理**する（飛ばさない・並行しない）。
2. ただし、**すでに完了しているコミットタスクは再実施しない**。

   * まず「進捗確認」を行い、**未完了の最小C-XXX番号**から再開する。
3. 各コミットタスクは必ず以下のステップで出力する。

   * **Step 1. 参照設計書の確認**（対象範囲、要件、制約、受け入れ条件、差分確認）
   * **Step 2. 事前確認**（既存コード/設定/規約、影響範囲、進捗確認）
   * **Step 3. 実装方針**（設計判断、代替案、採用理由、リスク）
   * **Step 4. 変更内容**（ファイル単位で、必要なコード/設定/SQLを提示）
   * **Step 5. 検証手順**（実行コマンド、期待結果、観点、チェックリスト）
   * **Step 6. 受け入れ基準の自己チェック**（基準項目に対する✅/❌と根拠）
4. **各コミットタスクに記載された「参照設計書」を必ず確認してから着手する**。設計書の要件・制約・受け入れ条件を実装判断の基準にする。
5. **変更は原則として対象タスクに必要な最小範囲に限定する**。追加が必要なら「追加理由」「代替案」「最小変更」を示す。
6. 既存の流儀（命名、例外、ログ、テスト、フォーマット、依存関係）を **先に読み取り、それに合わせる**。不明なら推奨デフォルトを採用し、理由を説明する。
7. 不確定要素がある場合：

   * 追加の質問は最小限にしつつ、作業は止めずに **仮定（Assumptions）** を置いて進める。
   * 後で確定情報が来たら、差分修正の手順を提示する。
8. 変更は **ビルド/テストが通る単位** で提示し、最後に動作確認の観点を必ず出す。
9. **進捗管理の更新**：

   * コミットタスクを完了したら、タスクリスト（tasks.md / task_list.md）内の該当C-XXX項目に **チェック（例：`[x]` / `✅`）** を付ける。
   * 形式はタスクリストの既存表記に合わせる（既存が`- [ ]`なら`- [x]`、既存が`✅`運用なら✅）。
   * 変更は **完了チェックと最小の補足（任意：コミットハッシュ、短いメモ）** のみに限定する。
10. **ターミナルでコマンドを実行する際は下記のテンプレに従って、ファイルにログを保存し、末尾200行のみを出力すること**
```powershell
Set-Location <project-path>
$log = "<logfile>.log"

<command> *>&1 |
  Out-File -FilePath $log -Encoding utf8

Get-Content $log -Tail 200
```

11. **巨大ログ/巨大ファイルは read_file で丸ごと読まないこと**
- 禁止：read_fileで全量読み／無制限出力
- 必須：検索→抜粋→必要なら周辺追加（200〜500行程度で上限）
  - 末尾: Get-Content <file> -Tail 200
  - 検索: Select-String -Path <file> -Pattern 'ERROR|FATAL|Exception|Traceback|panic' | Select-Object -Last 50
  - 周辺: Get-Content <file> -TotalCount $end | Select-Object -Skip ($start-1) > log_excerpt.txt
- 読むのは抜粋ファイル（log_excerpt.txt）のみ
12. **巨大なターミナル出力は禁止。出力が多いコマンドは必ずログに保存し、表示は末尾200行のみ：**
```
mvn clean compile *> mvn.log; Get-Content .\mvn.log -Tail 200; exit $LASTEXITCODE
```
13. 設計書で指定された ID および定数は、大文字・小文字・記号を含め、設計書の記載と完全に一致するように実装すること。

---

## 1. 入力

このコマンドは `/tasks.md [@task_list.md]` の形式で入力されます。

あなたは対象技術スタックのシニアエンジニアとして、渡された **コミットタスク ( C-000 )タスク一覧（tasks.md / task_list.md）** に従い実装・検証・自己チェックまで行います。

各コミットタスクには **参照設計書** が必ず記載されている前提です。実装前にその設計書を確認してください。

---

## 2. 進捗確認（開始前に必ず実施）

コミットタスク ( C-000 )-001から機械的に開始せず、必ず以下を行って **開始すべきコミットタスク ( C-000 )** を決める。

1. **タスクリストの状態確認**

   * tasks.md（および @task_list.md が指定されている場合はそれも）を読み、コミットタスク ( C-000 )ごとに以下を判定して一覧化する：

     * 完了（チェック済み / ✅ など明確な完了印がある）
     * 未完了（未チェック、TODO、未着手表記）
     * 不明（曖昧な表記・部分完了・メモのみ等）
   * さらに、各コミットタスク ( C-000 )に記載された **参照設計書** を確認し、目的・制約・受け入れ条件を整理する。

2. **リポジトリ状況の整合確認（可能な範囲で）**

   * 既存ブランチ、既存コミットタスク ( C-000 )/マージ済みコミット、差分、テスト状況などを確認し、
     「タスクリスト上は未完了だが実装済み」または「チェック済みだが未反映」の矛盾がないか確認する。
   * 矛盾がある場合は、止めずに仮定を置きつつ **最小確認で解消**する（例：該当ファイルの存在/差分/CIログの有無）。

3. **開始コミットタスク ( C-000 )の決定ルール**

   * 原則：**未完了の最小コミットタスク ( C-000 )番号** から開始。
   * ただし、以下の場合は「適切な開始地点」を調整してよい：

     * 連番の前提コミットタスク ( C-000 )が「実装は完了しているがチェックが未更新」：→ 先にチェック更新（根拠を提示）して次へ。
     * 「不明」判定のコミットタスク ( C-000 )がある：→ そのコミットタスク ( C-000 )から着手し、Step 2.事前確認で状況を確定させる。
   * いずれの場合も **番号順を崩さない**（必要なら、先に状態確定だけ行い、実作業は順番通り）。

4. **チェック更新（開始前の補正）**

   * 状況確認の結果、完了が明らかなコミットタスク ( C-000 )が未チェックなら、タスクリストにチェックを付ける（根拠も添える）。
   * 逆に、チェック済みだが未完了が明らかな場合は、チェックを外すのではなく、
     「不整合メモ」を追加してそのコミットタスク ( C-000 )から着手する（履歴を残す）。

## 3. コミットタスク ( C-000 )の進め方（コミットタスク ( C-000 )共通テンプレ）

### Step 1. 参照設計書の確認（必須）

* そのコミットタスク ( C-000 )に紐づく **参照設計書を、実装前に必ず読む。**
* 対象範囲、要件、制約、受け入れ条件、ID や定数を先に整理する。
* 既存実装と参照設計書の差分があれば、この時点で論点として明記する。

### Step 2. 事前確認（必須）

* コミットタスク ( C-000 )の目的と成功条件を1〜3行で要約する。
* **参照設計書** の対象箇所と、今回満たすべき要件を要約する。
* **進捗確認（このコミットタスク ( C-000 )が未完了である根拠 / もしくは完了済みならスキップする根拠）** を明記。

  * 完了済みなら：タスクリストにチェック更新し、**次のコミットタスク ( C-000 )へ**進む（B〜Eは省略可）。
* 既存実装の規約・流儀を調査して整理：

  * パッケージ構成 / 命名規則
  * 例外設計（例外階層、メッセージ、エラーコード有無）
  * ログ方針（ログレベル、PIIの扱い）
  * DB/ORM方針（型、NULL、FK、インデックス、監査、楽観/悲観ロック）
  * テスト方針（単体/統合、DB、CI）
* 影響範囲（参照設計書に紐づく実装箇所と関連する周辺）を列挙。

### Step 3. 実装方針（必須）

* 重要な設計判断を箇条書き：

  * 参照設計書との対応関係
  * データモデル/IF/ドメイン設計
  * 例外・バリデーション・トランザクション
  * 互換性（既存API/DBとの整合）
* 迷いどころがあれば「代替案」と「採用理由」を1つ以上書く。
* 重大なリスクがある場合、軽減策（ガード・テスト・ロールバック）を提示。

### Step 4. 変更内容（必須）

* 参照設計書の要件を満たすために変更したファイルごとに、**差分が分かる形で** コード/設定/SQLを提示。
* 新規ファイルは全文を提示。
* 設定値やSQLは、環境差分（dev/test/prod）に配慮し、既存方針に合わせる。
* 必要に応じて、

  * マイグレーション/DDLの整合
  * 依存関係追加
  * 例外・エラーレスポンス
  * ドキュメント（README、API仕様）

  も最小限で追加（ただし追加理由を明記）。

### Step 5. 検証手順（必須）

* ローカルでの検証コマンド例（ビルド、テスト、起動、lint/format等）。
* 期待結果（成功/失敗時の観察ポイント）。
* 追加で確認すべき観点（境界値、異常系、性能、並行実行、互換性）。

### Step 6. 受け入れ基準の自己チェック（必須）

* 受け入れ基準ごとに ✅ / ❌ を付ける。
* 判定の根拠を簡潔に記載する。

---

## 4. コミットタスク ( C-000 )完了時の追加アウトプット（任意だが推奨）

* コミットタスク ( C-000 )の要約（何をどう変えたか）
* レビュアー向け注意点（設計判断、互換性、マイグレーション注意）
* 次コミットタスク ( C-000 )に渡すメモ（依存関係、TODO、リスク）
* **タスクリスト更新結果**（どのコミットタスク ( C-000 )にチェックを付けたか／メモを追記したか）

---

## 5. 開始指示

1. まず **「2. 進捗確認」** を実施し、開始すべきコミットタスク ( C-000 )番号を宣言してください。
2. 次に、そのコミットタスク ( C-000 )の **参照設計書を実装前に読んでください。**
3. その後、そのコミットタスク ( C-000 )について **Step 1.参照設計書の確認 → Step 2.事前確認 → Step 3.実装方針 → Step 4.変更内容 → Step 5.検証手順 → Step 6.受け入れ基準の自己チェック** の順で完了させてください。
4. コミットタスク ( C-000 )完了後はタスクリストにチェックを付け、次のコミットタスク ( C-000 )へ進んでください。
5. すべて完了済みの場合は、完了報告とタスクリスト整合性（抜け漏れチェック）を提示して終了してください。
