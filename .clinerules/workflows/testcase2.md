# テストケース作成指示

## ユーザー入力
このコマンドは `/testcase.md [テスト対象]` の形式で入力されます。
受け取った [テスト対象] をテスト対象の機能としてください。

## 役割
あなたはQAエンジニアとして、手順にしたがってテストケースを作成すること。

## 進行ルール
- 各手順の成果物をユーザーに提示した後、**必ずユーザーの確認を待ってから**次の手順に進むこと
- ユーザーから修正指示があった場合は、その手順の成果物を修正し、再度確認を求めること
- ユーザーが承認した場合のみ、次の手順に進むこと
- 確認時には、現在の手順番号と次に進む手順を明示すること

## 手順

### 1. 要件を読む
まず `docs/` フォルダ内の設計書や関連資料から要件を読み込み、何を実現すべきか、何が正しい動作なのかを把握してください。
この段階では、対象機能の目的、前提条件、制約事項を明確にすることを重視してください。

**【確認】** 要件整理の結果（機能の目的・前提条件・制約事項・正しい動作の定義）をユーザーに提示し、以下の形式で確認を求めてください。
> 📋 **手順1: 要件整理が完了しました。**
> 上記の内容で問題なければ、次の手順（2. 業務フローに分ける）に進みます。
> 修正点があればお知らせください。

### 2. 業務フローに分ける
要件を確認したら、対象機能を業務フロー単位で整理してください。
利用者の操作や処理の流れを分解し、どの場面で何を確認すべきかが見える状態にしてください。

**【確認】** 業務フロー一覧をユーザーに提示し、以下の形式で確認を求めてください。
> 📋 **手順2: 業務フローの整理が完了しました。**
> 上記の内容で問題なければ、次の手順（3. テスト戦略を提案する）に進みます。
> フローの追加・修正・分割などがあればお知らせください。

### 3. テスト戦略を提案する
業務フローと要件を踏まえて、対象機能にとって**どのテスト種別を重点的に実施すべきか**を分析し、テスト戦略として提案してください。

#### 分析の進め方
1. **機能特性の分析**: 対象機能がどのような性質を持つか（ユーザー入力が多い、外部連携がある、データ更新を伴う、権限制御がある、など）を整理する
2. **リスク評価**: 不具合が発生した場合の影響度と発生可能性から、テスト重点領域を特定する
3. **テスト種別の選定**: 下記のテスト種別一覧から、対象機能に該当するテスト種別を選定する

#### テスト種別一覧
以下のテスト種別から、対象機能に関連するものを選定し、重点度を判定してください。

| テスト種別 | 適用場面 | 主な観点 |
| -------- | -------- | -------- |
| 機能テスト（バリデーション） | すべての機能 | 必須チェック、文字列長、数値範囲、文字種、相関バリデーション |
| 機能テスト（DB操作） | データ更新を伴う機能 | CRUD操作、トランザクション、排他制御、データ整合性 |
| 機能テスト（画面） | Web画面を持つ機能 | 画面表示、画面遷移、入力制御、レスポンシブ対応 |
| 機能テスト（バッチ） | バッチ処理 | 正常終了、異常終了、リカバリ、多重起動、スケジュール実行 |
| 機能テスト（API） | API・Webサービス | リクエスト/レスポンス、ステータスコード、認証、エラーハンドリング |
| データ互換性テスト | データ移行・連携がある機能 | データ変換、フォーマット互換、文字コード、欠損データ処理 |
| 業務シナリオテスト | 業務フローが複雑な機能 | 一連の業務操作、分岐フロー、例外フロー、業務ルール |
| セキュリティテスト | 認証・認可・入力処理がある機能 | インジェクション対策、認証・認可制御、セッション管理、XSS/CSRF対策、ログ出力 |
| 性能テスト | 高負荷が想定される機能 | レスポンスタイム、スループット、リソース使用率、スケーラビリティ |
| ストレス・ボリュームテスト | 大量データ・高頻度アクセスがある機能 | 高負荷時の挙動、大量データ処理、長時間稼働、リソース枯渇 |
| 障害テスト | 可用性・耐障害性が求められる機能 | 障害検知、フェイルオーバー、データ復旧、縮退運転 |
| 構成テスト | 複数環境・構成で動作する機能 | OS/ブラウザ差異、ミドルウェアバージョン、ネットワーク構成 |
| ユーザビリティテスト | エンドユーザーが直接操作する機能 | 操作性、わかりやすさ、エラーメッセージ、アクセシビリティ |

#### 提案フォーマット
以下の形式でテスト戦略を提案してください。

**🔍 機能特性の分析結果:**
（対象機能の特性を箇条書きで整理）

**⚠️ リスク評価:**
（不具合発生時の影響が大きい領域を特定）

**📊 テスト戦略（重点度の提案）:**

| テスト種別 | 重点度 | 該当理由 |
| ---------- | ------ | -------- |
| （種別名） | 高/中/低/対象外 | （なぜその重点度か） |

- **重点度「高」**: 必ず網羅的にテストすべき。ケース数を多めに確保する
- **重点度「中」**: 主要パターンをカバーする。代表的なケースを中心に実施する
- **重点度「低」**: 基本的な確認のみ実施する
- **対象外**: この機能では該当しないため省略する

**【確認】** テスト戦略の提案をユーザーに提示し、以下の形式で確認を求めてください。
> 📋 **手順3: テスト戦略の提案が完了しました。**
> 上記の重点度で問題なければ、次の手順（4. 観点を広げる）に進みます。
> 重点度の変更や、追加で考慮すべきリスクがあればお知らせください。

### 4. 観点を広げる
手順3で合意したテスト戦略（重点度）に基づいて、テスト観点を広げてください。

- **重点度「高」のテスト種別**: 正常系・異常系・境界値・分岐パターンなどを網羅的に洗い出す
- **重点度「中」のテスト種別**: 主要な正常系・異常系を中心に洗い出す
- **重点度「低」のテスト種別**: 基本的な正常系のみ確認する観点を洗い出す
- **対象外**: 観点の洗い出しをスキップする

手順3で選定した各テスト種別の「主な観点」を参考に、重点度に応じた粒度で観点を選定してください。

**【確認】** テスト観点一覧をユーザーに提示し、以下の形式で確認を求めてください。
> 📋 **手順4: テスト観点の洗い出しが完了しました。**
> 上記の内容で問題なければ、次の手順（5. シナリオ化する）に進みます。
> 観点の追加・削除・修正があればお知らせください。

### 5. シナリオ化する
洗い出した観点を、実際の利用場面がわかる形でシナリオにしてください。
「誰が、どのような条件で、何を行い、どうなるか」が一連の流れとして整理されている状態を目指してください。

**【確認】** テストシナリオ一覧をユーザーに提示し、以下の形式で確認を求めてください。
> 📋 **手順5: テストシナリオの作成が完了しました。**
> 上記の内容で問題なければ、次の手順（6. ケース化する）に進みます。
> シナリオの追加・修正があればお知らせください。

### 6. ケース化する
作成したシナリオを具体的なテストケースに落とし込んでください。
各テストケースには、少なくとも以下を明確に記載してください。

* 入力値
* 操作手順
* 期待結果

**【確認】** テストケース一覧をユーザーに提示し、以下の形式で確認を求めてください。
> 📋 **手順6: テストケースの作成が完了しました。**
> 上記の内容で問題なければ、ファイルに出力します。
> ケースの追加・修正があればお知らせください。

## 出力

- 出力先: `_testcase/testcase_<機能名>.md`
- ファイル名の `<機能名>` はユーザーが指定した機能名を英語のスネークケースに変換して使用すること
- 最終的なテストケース表だけでなく、要件整理、業務フロー、テスト観点、シナリオの中間成果も同じファイルに残すこと
- 中間成果は、テストケースの根拠が追跡できる粒度で簡潔に整理すること
- 出力は以下の構成に従うこと
