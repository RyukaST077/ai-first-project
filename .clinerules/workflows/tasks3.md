# 目的と役割

> 目的：実装計画書のタスクを **コミット単位（Atomic Commit）の作業タスクリスト** に分解し、チームが効率的に並行開発できる形で出力する。
> 役割：あなたは **シニア・エンジニア/スクラムマスター**。

---

## 入力パラメータ（必須）

* `TASK_ID`：実装計画書 `docs/implements_plan.md` に記載されているタスクID（例：`TASK-123`）

⚠️ `TASK_ID` が提示されていない場合は、**最初に `TASK_ID` の提示を求めて終了**する（タスクファイルは作成しない）。

---

## 対象ファイル

* `docs/implements_plan.md`（実装計画書）
* `docs/` 配下の設計書（仕様・I/F・制約の確認用）

作成したタスクファイルは `_tasks/<TASK_ID>.md` として出力してください。

---

## 絶対ルール

1. **設計書から最大限の情報を抽出する。** 設計書に記載された情報を優先して使用する。
2. **設計書にない仕様は `TBD` として明示する。** 推測で補完せず、ユーザーに質問して解消する。
3. **TBDが残る状態でタスクファイルを確定・実装に着手しない。** TBDがある場合は `_tasks/<TASK_ID>.md` を **ドラフト状態** で作成してよいが、すべてのTBDが解消されてから **タスクファイル確定・実装作業** に進む。
4. **1コミット = 1目的（Atomic Commit）** を厳守する。無関係な機能追加/リファクタ/テスト改善/ドキュメント整理/インフラ変更を混在させない。
   * 例外：**目的達成に必須な最小限のテスト/ドキュメント更新**は同コミットに含めてよい（目的が同一であること）。
5. **受け入れ基準はターミナルで自動検証可能** にする。`コマンド実行 → exit code / 出力` の形式で記述し、目視確認・ブラウザ確認・主観的判断を含めない。
6. **コミットはビルド/テストが通る単位** で刻む。中間状態で壊れたままにしない。
7. **コミットメッセージは Conventional Commits 形式** に従う。`<type>(<scope>): [<TASK_ID>] <description>` の形式を使用する。
8. **対象タスクのスコープを厳守する（Task Fence）。** 実装計画書の **他タスクに属する機能・画面・APIを実装しない**。
   * 他タスクで定義・実装される型・インターフェース・機能を **先行実装しない**（YAGNI原則）。
   * 「あると便利」な変更（スコープ外のリファクタ・コード整形・命名変更等）を行わない。
   * 依存タスクの成果物は **参照（import）のみ** 許可。変更・拡張はスコープ逸脱とみなす。
   * スコープ判断に迷う場合は **実装しない** 方を選び、TBDまたは境界メモとして記録する。
   * 各コミットの変更対象が「スコープ境界」セクションの IN SCOPE に含まれることを **事前に確認** する。

### スコープ逸脱の禁止事項（Negative Prompting）

以下の行為は **明示的に禁止** する：

| 禁止事項 | 理由 |
| --- | --- |
| 実装計画書に記載のない機能の追加 | スコープ外 |
| 他タスクの対象機能・画面・APIの実装 | 担当タスク違い |
| 将来のタスクを見越した先行実装 | YAGNI原則違反 |
| 他タスクが所有するファイルの変更 | オーナーシップ違反 |
| スコープ外のリファクタリング・コード整形 | 目的外の変更 |
| 「ついでに」行う改善・最適化 | スコープクリープ |
| OUT OF SCOPEに記載された項目の実装 | 境界違反 |

## 入力（あなたが最初にやること）

1. `docs/implements_plan.md` からタスクID `TASK_ID` を検索する
2. 該当タスクの以下の情報を抽出する：

   * 優先度（P0/P1/P2）
   * 対象（機能ID/画面ID/API ID等）
   * 依存関係（前提タスク）
   * 内容（ざっくり）
   * 方式/観点（例：新規実装/改修/バグ修正/設定変更/移行/品質/セキュリティ など）
3. **隣接タスクの分析（スコープ境界の定義）**
   * 実装計画書から対象タスクの **依存元・依存先・同一フェーズのタスク** を抽出
   * 各隣接タスクの「対象」「内容」を確認し、境界を定義
   * IN SCOPE / OUT OF SCOPE / BOUNDARY テーブルを作成
4. `docs/` 配下の設計書から、実装に必要な仕様・制約・I/F・データ構造を確認する

## 最初の回答（必須）

最初の回答は必ず以下を含める：

* **(a) タスク情報の要約**

  * タスクID、対象、概要
  * 依存関係（前提/後続）

* **(b) 参照した設計書と抽出した仕様**

  * 関連する機能ID/画面ID/API ID
  * 実装に必要な仕様・制約

* **(c) 不足情報（TBD）一覧**

  * 設計書に記載がない項目（`TBD` として記録）
  * 前提条件チェックリストへの反映

* **(d) スコープ境界**

  * IN SCOPE（このタスクで実装すること）
  * OUT OF SCOPE（他タスクの範囲であり実装しないこと）
  * BOUNDARY（参照のみ許可の共有リソース）

* **(e) コミット分割案**

  * コミット数と各コミットの概要（`C-001`〜`C-020`）
  * 依存関係（直列必須 / 並行可能を明示）

## 重要なルール（必須）

### 1) タスクIDが存在しない場合

`docs/implements_plan.md` に **タスクIDが存在しない場合**：

1. **`_tasks/<タスクID>.md` は作成しない**
2. 以下を出力して終了する：

   * 「タスクID `TASK_ID` が見つかりません」
   * 追加で必要な情報（質問項目）を箇条書きで提示
   * `docs/implements_plan.md` への追加提案（必要に応じて）

### 2) 設計情報が不足している場合

`docs/` に必要な設計情報が不足している場合：

1. 不足点は **`TBD`** として一時的に記録する
2. 関連しそうな設計書を追加で検索し、情報を補完する
3. それでも不足する場合は **ユーザーに質問して解消する**
4. **すべてのTBDが解消されてから** タスクファイル確定・実装作業に進む

### 3) コミット分割ルール（Atomic Commit）

| ルール             | 説明                                          |
| --------------- | ------------------------------------------- |
| **1コミット = 1目的**   | 無関係な機能追加/リファクタ/テスト改善/ドキュメント整理/インフラ変更を混在させない |
| **依存関係の明記**     | 後続コミットの「前提条件」に `C-XXX` を明記                   |
| **採番ルール**       | `C-001` から連番、**最大 `C-020`** まで（超える場合は統合）  |
| **受け入れ基準**      | ターミナルで実行可能なコマンドで記述（exit code / 出力で検証・目視確認禁止） |
| **前提条件は自動検証可能** | コマンド実行またはファイル確認で検証できる形式で記述（ユーザー確認不要）        |

#### 前提条件の記述ルール（重要）

前提条件は **AIエージェントがコマンド実行またはファイル確認で自動検証できる形式** で記述する。ユーザーへの確認が必要な条件は前提条件に含めない。

| 種別               | 検証方法                                 | 記述例                                                            |
| ---------------- | ------------------------------------ | -------------------------------------------------------------- |
| **ファイル存在**       | `ls` / `test -f`                     | `src/types/user.ts` が存在すること                                    |
| **ディレクトリ存在**     | `ls` / `test -d`                     | `src/components/` ディレクトリが存在すること                                |
| **コマンド実行可能**     | `which` / `command -v`               | `npm` がインストールされていること                                           |
| **テスト通過**        | `npm test` / `pytest`                | `npm test` が成功すること（exit code 0）                                |
| **ビルド成功**        | `npm run build`                      | `npm run build` が成功すること                                        |
| **型チェック通過**      | `npm run typecheck` / `tsc --noEmit` | 型エラーがないこと                                                      |
| **Lint通過**       | `npm run lint`                       | Lintエラーがないこと                                                   |
| **依存関係インストール済み** | `package.json` 確認                    | `package.json` に `zod` が含まれていること                               |
| **環境変数設定済み**     | `.env` / `.env.example` 確認           | `.env.example` に `DATABASE_URL` が定義されていること                     |
| **コミット完了**         | `_tasks/<タスクID>.md` のチェック状態          | C-002 の受け入れ基準がすべてチェック済みであること                                  |
| **設計書存在**        | ファイル確認                               | `docs/02_Detailed_Design/08_Interface_Design/` に該当API定義が存在すること |
| **スキーマ定義済み**     | ファイル内容確認                             | `prisma/schema.prisma` に `User` モデルが定義されていること                  |
| **マイグレーション適用済み** | `prisma migrate status`              | 未適用のマイグレーションがないこと                                              |

**❌ 禁止される前提条件の例:**

* 「ユーザーが仕様を確認済みであること」（ユーザー確認が必要）
* 「チームで合意が取れていること」（主観的な判断が必要）
* 「設計が完了していること」（曖昧、何をもって完了か不明）
* 「準備ができていること」（検証方法が不明）

#### 受け入れ基準の記述ルール（重要）

受け入れ基準の「検証方法」は **ターミナルで実行可能なコマンド** のみ使用する。目視確認・ブラウザ確認・主観的判断は禁止。

| 種別 | 検証コマンド | 期待結果の例 |
| ---- | ----------- | ----------- |
| **ビルド成功** | `npm run build` / `cargo build` | exit code 0 |
| **テスト全通過** | `npm test` / `pytest` / `cargo test` | exit code 0、失敗0件 |
| **型チェック通過** | `npx tsc --noEmit` / `npm run typecheck` | exit code 0、エラー0件 |
| **Lint通過** | `npm run lint` / `npx eslint src/` | exit code 0 |
| **ファイル生成確認** | `test -f dist/index.js && echo ok` | `ok` が出力される |
| **APIエンドポイント応答** | `curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/api/xxx` | `200` |
| **HTTPレスポンスボディ確認** | `curl -s http://localhost:3000/api/xxx \| jq '.key'` | 期待する値 |
| **DBマイグレーション適用済み** | `npx prisma migrate status` | `No pending migrations` |
| **特定クラス/関数の存在** | `grep -rn "ClassName" src/` | 1件以上ヒット |
| **エラーログ未出力** | `npm test 2>&1 \| grep -c ERROR` | `0` |
| **カバレッジ閾値** | `npm run test:coverage` | 閾値（例：80%）以上 |

**❌ 禁止される受け入れ基準の例:**

* 「ブラウザで画面を確認する」（自動化不可）
* 「目視で動作を確認する」（自動化不可）
* 「正常に動作すること」（曖昧、観測方法が不明）
* 「エラーが出ないこと」（コマンドなし）
* 「ユーザーが承認すること」（主観的判断）

## コミット要件

| 項目      | ルール                                    |
| ------- | -------------------------------------- |
| メッセージ形式 | `<type>(<scope>): [<タスクID>] <description>`（Conventional Commits） |
| 実装コミット  | ビルド/テストが破綻しない単位で刻む                     |

### Conventional Commits タイプリファレンス

| type | 用途 | 例 |
| ---- | ---- | --- |
| `feat` | 新機能追加 | `feat(user): [TASK-123] Add registration API` |
| `fix` | バグ修正 | `fix(auth): [TASK-123] Fix token expiration check` |
| `refactor` | リファクタリング（機能変更なし） | `refactor(db): [TASK-123] Extract query builder` |
| `docs` | ドキュメントのみの変更 | `docs(design): [TASK-123] Update API spec` |
| `test` | テストの追加・修正のみ | `test(user): [TASK-123] Add unit tests for UserService` |
| `chore` | 雑務（タスクファイル作成等） | `chore(tasks): [TASK-123] Create task file` |
| `ci` | CI設定変更 | `ci(github): [TASK-123] Add lint workflow` |
| `style` | フォーマット変更のみ | `style: [TASK-123] Apply prettier` |
| `perf` | パフォーマンス改善 | `perf(query): [TASK-123] Add index for user lookup` |
| `build` | ビルドシステム・依存関係変更 | `build: [TASK-123] Update dependencies` |

## 進め方（フェーズ制・ゲート方式）

### フェーズ0：タスク情報の確認

1. `docs/implements_plan.md` からタスクIDを検索
2. タスク情報（優先度/対象/依存関係/方式・観点）を抽出
3. 関連する設計書を特定

**確認ゲート0（自動検証）**

| チェック項目        | 検証方法                                      | 期待結果                         |
| ------------- | ----------------------------------------- | ---------------------------- |
| タスクIDが存在する    | `grep -n "$TASK_ID" docs/implements_plan.md` | 1件以上ヒット                      |
| 依存関係が記載されている  | 該当タスク行の「依存」列を確認                           | 空欄でない or 「なし」と明記             |
| 分割戦略の根拠が確認できる | 該当タスク行または設計書の記載を確認                        | `split_strategy` の選定理由が記述できる |

---

### フェーズ0.5：スコープ境界の定義（隣接タスク分析）

1. 実装計画書から対象タスクの **依存元・依存先・同一フェーズのタスク** を抽出
2. 各隣接タスクの「対象」「内容」を確認し、境界ルールを定義
3. IN SCOPE / OUT OF SCOPE / BOUNDARY テーブルを作成
4. 共有リソース（複数タスクが触るファイル/モジュール）のオーナーシップを明確化

**スコープ判断フロー（各変更に対して適用）**

```
1. この変更は TASK_ID の受け入れ基準を満たすために必要か？
   → No → 実装しない
2. この変更は受け入れ基準の達成に最小限必要か？
   → No → 実装しない（将来の予防的追加・便宜的追加はすべて除外）
3. この変更は他のTASK_IDで実装される予定か？
   → Yes → 実装しない（そのタスクに委ねる）
4. 上記すべてクリア → 実装する
```

**確認ゲート0.5（自動検証）**

| チェック項目 | 検証方法 | 期待結果 |
| --- | --- | --- |
| IN SCOPEが定義されている | タスクファイルの「スコープ境界」セクションを確認 | IN SCOPEに1件以上の項目が記載 |
| OUT OF SCOPEが定義されている | 隣接タスクから除外項目を抽出 | 1件以上の除外項目が記載（担当タスクID付き） |
| BOUNDARYが定義されている | 共有リソースの権限を確認 | 各共有リソースに権限（新規作成/変更/参照のみ）が明記 |
| 隣接タスクの対象と重複がない | IN SCOPEと隣接タスクの対象を照合 | 重複する機能ID/画面ID/API IDがない |

⚠️ **スコープ境界が未定義の場合、フェーズ1に進まない**

---

### フェーズ1：設計情報の収集

1. `docs/` 配下から関連する設計書を読み込む
2. 実装に必要な仕様・制約・I/F・データ構造を抽出
3. 不足情報を特定

**確認ゲート1（自動検証）**

| チェック項目             | 検証方法                                | 期待結果          |
| ------------------ | ----------------------------------- | ------------- |
| 関連設計書が存在する         | `ls docs/` で該当ファイルを確認               | 必要なファイルがすべて存在 |
| 設計書に該当IDが記載されている   | `grep "<機能ID/画面ID/API ID>" <設計書パス>` | 1件以上ヒット       |
| 不足情報がTBDとして記録されている | 作成中のタスクファイルを確認                      | TBDセクションに記載済み |

---

### フェーズ1.5：TBD解消（ユーザーへの質問）

TBDが存在する場合、このフェーズでユーザーに質問して解消する。

1. TBD一覧をユーザーに提示する
2. 各TBDについて具体的な質問を行う（選択肢を提示）
3. ユーザーの回答を受けてTBDを解消する
4. **解消した内容を設計書に反映する**（該当する `docs/` 配下のファイルを更新）
5. すべてのTBDが解消されたらフェーズ2に進む

**確認ゲート1.5（自動検証）**

| チェック項目        | 検証方法                    | 期待結果                   |
| ------------- | ----------------------- | ---------------------- |
| すべてのTBDが解消された | タスクファイルのTBDセクションを確認     | TBD一覧が空 or すべて「解消済み」   |
| 設計書に反映された     | `git diff docs/` で変更を確認 | TBD解消内容が設計書に追記されている    |
| タスクファイルに反映された | タスクファイルの該当箇所を確認         | TBDだった項目が具体的な値に置換されている |

⚠️ **TBDが残っている場合、フェーズ2に進まない**

---

### フェーズ2：コミット分割の設計

1. **`task_type` と `split_strategy` の確定**

   * 計画書/設計書のキーワードと観点（新規/改修/移行/品質/セキュリティ等）を確認
   * `split_strategy` を enum から1つ選定し、理由を書く

   > 注：`task_type` はタスク全体の性質（例：実装・リファクタ）。`split_strategy` はコミット分割の切り口（例：new_feature・refactor）。両者は異なる観点だが一致する場合もある。

2. **コミット分割の設計（一般ルール）**

   * 依存が強い順に「土台 → 中核 → 結合 → 仕上げ」で切る
   * 1コミット＝1論理変更（Atomic Commit）を守る（目的に必須な最小限のテスト/ドキュメントは同コミット可）
   * コミットが20を超えそうなら統合して調整
   * 各コミット単独でビルド/テストが通る状態を維持する

3. **コミット間の依存関係を整理**

   * 前提条件を明記
   * 直列必須 / 並行作業可能なコミットを識別

**確認ゲート2（自動検証）**

| チェック項目              | 検証方法                                | 期待結果                  |
| --------------------- | ----------------------------------- | --------------------- |
| 分割戦略を正しく判定した          | タスクファイルに `split_strategy: X` が記載    | enumのいずれか              |
| 1コミット = 1目的になっている      | 各コミットの「目的」セクションを確認                  | 各コミットに単一の目的が記載         |
| コミット数が最大20個以内          | `grep -c "^## C-" <タスクファイル>`         | 20以下                   |
| 依存関係が正しく設定されている       | 各コミットの前提条件テーブルを確認                   | 依存コミットが前提条件に記載         |
| 受け入れ基準が自動検証可能 | 受け入れ基準の「検証コマンド」列を確認 | ターミナルコマンドのみ（目視確認なし） |
| **全コミットがIN SCOPE内である** | 各コミットの目的・変更ファイルとIN SCOPEを照合 | OUT OF SCOPEに触れるコミットがない |
| **BOUNDARY権限に違反がない** | 変更予定ファイルとBOUNDARYテーブルを照合 | 「参照のみ」ファイルを変更するコミットがない |
| **各コミットがYAGNI原則に従っている** | 各コミットの実行タスクと変更ファイルを確認 | 受け入れ基準の達成に不要な変更を含まない |
| **他タスクの変更を含まない** | 各コミットの変更ファイルと隣接タスクの対象を照合 | 他タスク所有ファイルへの変更がない |

---

### フェーズ3：タスクファイルの作成

1. テンプレートに沿ってタスクファイルを作成
2. 受け入れ基準の「検証コマンド」列にターミナルで実行可能なコマンドを記載する（目視確認・ブラウザ確認は記載しない）

**確認ゲート3（自動検証）**

| チェック項目            | 検証方法               | 期待結果              |
| ----------------- | ------------------ | ----------------- |
| すべてのセクションが埋まっている  | タスクファイルの各セクションを確認  | 「TBD」や空欄がない       |
| 受け入れ基準が自動検証可能       | 各受け入れ基準の「検証コマンド」列を確認 | ターミナルコマンドのみ・目視確認なし |
| 前提条件が自動検証可能       | 各コミットの前提条件テーブルを確認    | すべての前提条件に検証方法が記載  |
| **スコープ境界セクションが完備** | IN/OUT/BOUNDARYがすべて埋まっている | 空行・プレースホルダがない |
| **全コミットのスコープ検証済み** | フェーズ2の確認ゲート2のスコープ項目がすべてパス | OUT OF SCOPE・BOUNDARY違反・YAGNI違反がない |
