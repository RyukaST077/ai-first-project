# テスト工程

## 画面テスト

実装が終わった段階で画面のテストを行います。テストの実行は各画面の担当エンジニアがテストを実行します。
流れとしては `テスト計画を作成→人間が計画をレビュー→テスト用データを作成→テストを実施` です。

テストの計画とデータ作成を先に行うことでテストケースの抜けもれを減らし、品質を安定させるためのテストの実行ができます。

### Step1. 画面テスト計画書の作成

1. Cline へ *`/make_screen_testcase.md [対象のテスト画面名]`* と送信。
2. `docs/test/screen/` フォルダ内に対象画面のテスト計画書が生成されていること、内容を確認。

        `ctrl + shift + v` で見やすく表示できます。
3. 不足している観点や、重視してほしい観点があれば同じ会話の中で追加指示をする。


### Step2. テスト用データの生成
AIとともにどのようなテストデータを用意するべきかを考えることで、テストケース毎にテストデータを用意するよりも網羅的で品質の高いテストを行うことが可能です。

1. Cline へ *`/think_test_data.md [先ほど作成したテスト計画書パス]`* と送信。

    このカスタムコマンドは、テスト計画書を元に必要なテストデータの計画をしてくれます。
2. AIから下記の出力がされます。出力に対して、必要なデータの修正指示や疑問点など、必要に応じて会話する。
    
    - 必要なマスタデータ一覧
    - テーブル毎の作成データ案
    - テストケースとの対応表
    - 不足・曖昧点

3. 修正点の指示や、不明点の解消が終わったら、 *`/create_test_data.md`* と送信。
4. 生成された sqlファイル を使ってデータベースへINSERTしたりしてデータを用意する。
    
    spring frameworkだったら、resourceフォルダ内に入れて初期化など、

### Step3. 生成されたテストケースとデータを使ってテストの実行
1. 画面テスト計画書マークダウンファイルの `## 3. テスト項目一覧` の内容をコピーし、microsoft copilot へ下記のようにプロンプトを打って送信。
    ````
    ## 依頼
    マークダウンテキストをもとに完全なエクセルファイルを生成すること

    ## マークダウンテキスト
    ```
    <生成したマークダウンファイルの内容をコピー & ペースト>
    ```
    ````
    これでエクセルファイルを使ってテスト結果の記録ができます。

2. 生成されたテスト計画書を元にテストを実行する。
    必要があればテスト実行済みのステータスカラムを追加したりしてテストケースを管理。
