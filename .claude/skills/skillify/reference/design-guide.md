# 良い Skill の設計原則

Skill を書くときの原則と、典型的なアンチパターン。Phase 5（ドラフト生成）で参照する。

---

## 1. description が最重要

`frontmatter.description` は **将来の Claude が「いま、この Skill を起動すべきか」を判断する唯一の情報源**。ここが曖昧だと誤起動・起動漏れが起きる。

### 良い description の要素

1. **何をする Skill か**（1 文で）
2. **典型的なユーザ発話** を 2〜3 例（「～と言ったら起動」）
3. **使わない場面** を 1 例（「～なら別の手段を使う」）
4. **前提条件** があれば明記

### 悪い例

```yaml
description: DB 関連のヘルパー
```

→ 何をするか、いつ呼ぶかが不明。類似タスク全部にマッチしてしまう。

### 良い例

```yaml
description: Laravel プロジェクトで、マイグレーションファイル追加前に既存スキーマとの整合性を自動検証する Skill。ユーザが「マイグレーションの妥当性チェック」「新しいマイグレーションを追加したい」と言ったら起動。既存マイグレーションの修正時や schema.rb を直接触る場合には使わない。前提: artisan が動作し、DB 接続可能であること。
```

---

## 2. Progressive Disclosure（情報の段階的開示）

SKILL.md 本体は **短く保つ**（200 行以下が目安）。長い補足は `reference/*.md` に分離する。

### 判断基準

| 情報の種類 | 配置先 |
|----------|-------|
| 起動条件、目的、主要手順 | SKILL.md |
| 具体的なコマンド、確認ポイント | SKILL.md |
| 失敗時の詳細な対応策 | `reference/troubleshooting.md` |
| 背景・設計思想 | `reference/design-notes.md` |
| 実行スクリプト（10 行以上） | `scripts/*.sh` |
| 生成ファイルの雛形 | `templates/*` |
| FAQ / よくある質問 | `reference/faq.md` |

---

## 3. 「使わない場面」を明記する

Skill のスコープを **ネガティブ側からも定義** する。起動すべきでないケースを書くことで誤起動が大幅に減る。

```markdown
**この Skill を使わないケース:**
- 既存ファイルの軽微な編集だけで済む場合（直接 Edit）
- 別の `X-skill` で扱うべきケース（～を参照）
- 本番環境への操作（手動承認フローが必要）
```

---

## 4. 起動条件は「具体的な発話例」で書く

抽象的な条件ではなく、**ユーザが実際に言いそうなフレーズ** を書く。

### 悪い例

> ユーザが DB 関連のバリデーションを求めた場合に起動する

### 良い例

> 次のいずれかが発話されたら起動:
> - 「マイグレーションチェックして」
> - 「DB スキーマ妥当性を確認」
> - 「この migration 追加しても大丈夫か見て」

---

## 5. 手順は「何を確認するか」までセットで

各 Step に **確認ポイント** を書く。単にコマンドを羅列するだけでは、失敗時に Claude が立ち往生する。

```markdown
## Step 2: マイグレーションを dry-run

```bash
php artisan migrate --pretend
```

**確認ポイント:**
- Nothing to migrate. が出たら Step 3 へ
- SQL 文が出力されたら内容を読み、Step 3 に進む前にユーザ承認を得る
- エラーが出たら失敗時対応 A へ
```

---

## 6. 抽象化しすぎない

**再利用性を優先するあまり、具体性を失わないこと**。

### 悪い例

```markdown
1. 適切なコマンドを実行する
2. 結果を検証する
3. 必要なら修正する
```

→ 情報量ゼロ。Claude はまた会話中に全部考え直す。

### 良い例

```markdown
1. `docker compose up -d db` で DB を起動
2. `curl http://localhost:8000/api/health` が 200 を返すことを確認
3. 200 以外なら `docker compose logs db` で原因特定
```

パラメータ化が必要な部分だけ `<var>` にする。

---

## 7. Skill化しないほうがよいもの

- **一度しかやらないこと** → CLAUDE.md に書くか、単に実行する
- **プロジェクト固有の知識そのもの** → CLAUDE.md に書く
- **非常に短い 1 ステップ作業** → CLAUDE.md のスニペット例で十分
- **判断の方針だけで、具体的手順がない** → CLAUDE.md の guideline 節へ

---

## 8. テスト可能に書く

Skill を書いた後、**別セッションで** トリガーフレーズを唱えて動作確認する。

- 期待通りに起動するか？
- 前提条件の確認を飛ばしていないか？
- 途中でユーザ承認を求めるべき箇所で立ち止まっているか？

---

## 9. description の長さ目安

- 100〜400 文字（日本語換算）
- 長すぎると skill listing の budget を圧迫する（`skillListingBudgetFraction` 設定参照）
- 短すぎると起動判定がブレる

---

## 10. バージョン管理

Skill を追加したら git に入れるか、`.gitignore` で除外するかを決める:

- **チームで共有したい** → `.claude/skills/` ごとコミット
- **個人用** → `.gitignore` に追加
- **秘密情報を含む** → 絶対にコミットしない（そもそも書かない）

今回（skillify で生成する）Skill はデフォルトでチーム共有を想定。
