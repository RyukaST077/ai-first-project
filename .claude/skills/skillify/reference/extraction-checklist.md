# 会話抽出チェックリスト（Phase 2 用）

現在の会話コンテキストから次の項目を順に抽出する。抽出したら Phase 2 のサマリとしてユーザに提示する。

## 1. 目的 / ゴール

- この作業で **最終的に達成したこと** は何か？
- ユーザが最初に持ち込んだ問いは何だったか？（会話冒頭のユーザ発話が手がかり）
- それがどう変化したか？（途中でゴールが変わっていないか）

## 2. 前提条件

- 作業開始時に **すでに満たされていた前提** は何か？
  - 必要なファイル、インストール済みツール、起動中サービス、git 状態、etc.
- その前提が崩れたら Skill はどの Step で失敗するか？

## 3. 実行した手順（時系列）

会話中の Bash / Write / Edit / Read 等のツール呼び出しを見て、実際に実行した順序を列挙:

| 順 | 操作 | 対象 | 目的 |
|----|------|------|------|
| 1 | Bash | `cmd ...` | ～を確認 |
| 2 | Edit | `path/to/file` | ～を修正 |
| 3 | Bash | `cmd ...` | ～を実行 |

**注意**: 「失敗した試行」「脇道」は Skill に含めない。最終的にうまくいったパスだけを抽出する。

## 4. 途中の意思決定

- どこで選択肢が分岐したか？
- どの選択肢を採用したか、採用理由は何か？
- **別プロジェクトで再利用するとき** その判断は毎回必要か、固定でよいか？

## 5. つまずき・罠

会話で「～というエラーが出た」「～だと思ったら違った」「実は～が必要だった」といった箇所をピックアップ。Skill の「注意点」セクションに入れる。

## 6. 使用したコマンド・ツール

- シェルコマンド
- 外部 CLI ツール（docker, kubectl, gh, etc.）
- Claude Code の内部ツール（Bash, Edit, Agent, etc.）
- MCP サーバ

これらが **前提条件** として必要かを判断。

## 7. 成果物

作業終了後に生まれた / 変わったもの:

- 作成ファイル、編集ファイル
- DB の状態変化
- デプロイ、PR、issue
- 確認結果（レポート、ログ）

## 8. スコープ確認（重要）

会話で実施したが **Skill に入れるべきではない** もの:

- 失敗した試行・ボツ案
- 本題から逸れた雑談
- 一度きりのデバッグ
- このプロジェクトにしか意味がない詳細情報（それはそれで残すが、注記をつける）

## 抽出サマリのフォーマット例

```
【目的】
APIサーバのバージョンアップ前に、エンドポイント一覧とレスポンススナップショットを確定するための初期セットアップを行った。

【前提】
- Laravel プロジェクトの routes/ が存在
- Docker Compose で DB が起動可能

【主な手順】
1. routes/*.php からエンドポイント抽出
2. tools/snapshot/ 一式を tools/ 配下に配置
3. capture.sh をプロジェクトに合わせて実装
4. 初回 snapshot を record → 健全性チェック → approve

【使用ツール/コマンド】
- bash tools/snapshot/snapshot.sh {record,test,approve}
- Docker Compose
- grep / awk（エンドポイント抽出）

【注意点】
- masker が効いていないと差分が無限に出る → Step 3 で必ず確認
- 初回 approve は盲目的にやらない（仕様妥当性を判定する）

【再利用時に変わる部分】
- プロジェクト固有のエンドポイント定義箇所
- DB リセット手順
```
