---
name: skillify
description: 現在の Claude Code 会話で実施した作業を、再利用可能な Skill（.claude/skills/<name>/）として切り出すための Skill。会話コンテキストからの自動抽出 → ユーザへの対話ヒアリング → 名前・構造の提案 → ドラフト生成 → 承認 → ファイル書き出し、までを一連でガイドする。ユーザが「この会話での作業をSkill化して」「いまやった作業をSkillにして」「この手順を再利用可能にして」などと言ったら起動する。新規 Skill 作成専用（既存 Skill の編集では使わない）。
---

# 目的

**いま Claude と一緒に行った作業** を、次回以降の別会話からでも同じ手順で呼び出せる **再利用可能な Skill** に昇格させる。

対象となる「作業」の典型例:
- 複数ステップの調査・検証フロー（例: DB 接続確認 → マイグレーション走らせ → smoke test）
- よくやる定型リファクタ
- プロジェクト固有の運用手順（デプロイ前チェック、ログ調査、等）
- 外部ツールやコマンドを組み合わせたワークフロー

**このSkillが作らないもの**:
- 一度しかやらないタスク（Skill化する価値なし）
- 既存 Skill の編集・改修（直接ファイルを編集すべき）
- プロジェクトをまたぐ汎用 Skill（ユーザレベル `~/.claude/skills/` 配置は別途検討）

---

# 起動条件

ユーザが次のいずれかを言ったら本 Skill を呼ぶ:

- 「この会話での作業をSkill化して」
- 「いまやった作業をSkillにして / 再利用可能にして」
- 「この手順をSkill化」
- 「さっきの一連の流れをSkillにまとめて」

**前提**:
- 現在の会話で **何らかのまとまった作業** が実行済み（もしくは設計・議論済み）
- プロジェクトに `.claude/skills/` が存在する（なければ作成）
- `.claude/settings.local.json` の `sandbox.filesystem.allowWrite` に `./.claude/skills/**` が含まれる（サンドボックス環境の場合）

---

# 全体フロー

```
Phase 1: 事前確認        → .claude/skills/ 準備、名前衝突回避の下準備
Phase 2: 抽出            → 現在の会話から作業の骨子をドラフト
Phase 3: ヒアリング      → 不足情報・抽象化ポイントを対話で補完
Phase 4: 名前・構造の提案 → 命名候補3つ + ディレクトリ構造を提示
Phase 5: ドラフト生成    → SKILL.md + 付随ファイルを画面に全文表示
Phase 6: 承認            → ユーザが OK / 修正依頼 / キャンセル
Phase 7: 書き出し        → ファイル作成、使い方案内
```

各 Phase はユーザ承認で次へ進む。**Phase 6 を完了するまでディスクに一切書き出さない**。

---

# Phase 1: 事前確認

Bash/Glob で次を確認:

1. `.claude/skills/` が存在するか（なければ作成を提案）
2. 既存 Skill 一覧（命名衝突チェック用）:
   ```bash
   ls -1 .claude/skills/ 2>/dev/null
   ```
3. サンドボックス write 許可の確認（必要に応じて `.claude/settings.local.json` の `sandbox.filesystem.allowWrite` を確認）

衝突しそうな名前があれば Phase 4 で避ける。

---

# Phase 2: 抽出（Claude 自動処理）

現在の会話コンテキストを読み返し、以下を **内部で** ドラフトする（まだユーザには見せない）。

→ 抽出すべき項目の詳細は [reference/extraction-checklist.md](reference/extraction-checklist.md) 参照。

主な抽出観点:
- **目的**: この作業は何を達成したか？
- **前提条件**: 何が揃っている必要があるか？
- **手順**: 実行した or 議論したステップ（コマンド・ファイル・ツール含む）
- **判断・意思決定**: 途中で分岐・検討した箇所
- **つまずき・罠**: ハマった点、避けるべきこと
- **成果物**: 最終的に何が生まれたか

抽出後、ユーザに **1 画面に収まる要約** を提示する:

```
抽出された作業の骨子（ドラフト）:

【目的】 ～
【前提】 ～
【主な手順】
  1. ～
  2. ～
  3. ～
【使用ツール/コマンド】 ～
【注意点】 ～

この内容で Phase 3（ヒアリング）に進んでよいですか？
追加・修正があれば教えてください。
```

---

# Phase 3: ヒアリング（対話）

Phase 2 のドラフトで **不足・曖昧な点を必ず質問** する。

→ 質問テンプレは [reference/interview-questions.md](reference/interview-questions.md) 参照。

必須確認事項:

1. **再利用時に変わる部分（パラメータ）**
   - ファイルパス、テーブル名、ブランチ名、対象ユーザー、…
   - これらは Skill 本文で `<var>` のように抽象化する

2. **再利用時に変わらない部分（固定）**
   - プロジェクト固有のコマンド体系、決まったチェック項目、…

3. **起動条件（trigger phrases）**
   - 次にこの Skill をどんな言葉で呼び出したいか
   - frontmatter の `description` に反映される重要項目

4. **スコープの確認**
   - 「会話の中で脇道に入った部分」を Skill に入れるか？
   - 本題と無関係な試行錯誤は除外する

5. **付随スクリプト/テンプレの要否**
   - 手順中の長いコマンドを `scripts/*.sh` として切り出すか？
   - 生成するファイルの雛形を `templates/` に置くか？

質問は **1 メッセージに複数まとめて** 聞く（往復を減らす）。

---

# Phase 4: 名前・構造の提案

Claude が **Skill名の候補を 3 つ提案** する。

命名ルール:
- 動詞 or 動詞句（何をする Skill かが一目でわかる）
- 英小文字 + ハイフン、簡潔（2〜3語）
- 既存 Skill 名と衝突しない
- 動作が具体的にイメージできる（汎用名 `helper`, `tool` は避ける）

提示例:
```
Skill名の候補:
  1. migrate-check       — DB マイグレーション検証フローを実行
  2. verify-migration    — マイグレーション妥当性を事前検証
  3. db-preflight        — DB 変更前の健全性チェック

どれが近いですか？ または別名を提示しますか？
```

合わせて **ディレクトリ構造案** も提示:

```
.claude/skills/<chosen-name>/
├── SKILL.md                  # 必須
├── scripts/                  # （任意）長いコマンドをまとめる場合
│   └── check.sh
├── templates/                # （任意）生成ファイルの雛形
│   └── ...
└── reference/                # （任意）詳細ガイド/FAQ など
    └── ...
```

**必要最小構成でスタート**。迷ったら SKILL.md 単独でよい。

---

# Phase 5: ドラフト生成

Phase 1〜4 で確定した内容から、**実際に書き出すファイルの全文** を対話画面に提示する。

ドラフトに含める要素:

1. **SKILL.md**（[templates/skill-template.md](templates/skill-template.md) ベース）
   - frontmatter: `name`, `description`
   - 目的 / 起動条件 / 前提 / 手順 / 注意点 / 成果物

2. **付随ファイル**（Phase 3 で合意された場合のみ）
   - `scripts/*.sh`
   - `templates/*`
   - `reference/*.md`

ディスクには **まだ書かない**。画面上で全文レビューしてもらう。

**良い SKILL.md の設計原則** は [reference/design-guide.md](reference/design-guide.md) を参照。description の書き方 / progressive disclosure / 起動条件の明示 / 否定条件（「この Skill を使わない場合」）の 4 点が特に重要。

---

# Phase 6: 承認

ユーザの応答:

| 応答 | 次アクション |
|------|-------------|
| 「OK」「これで」「書いて」 | Phase 7 へ |
| 「ここを直して」「～を追加」 | 指摘を反映し Phase 5 を再提示 |
| 「キャンセル」「やめる」 | 何も書かずに終了 |

---

# Phase 7: 書き出し & 完了案内

1. ディレクトリと全ファイルを `Write` ツールで作成
   - パス: `/home/kamayama/02_work/002_version_up_test/.claude/skills/<name>/`
   - **絶対パスを使う**（サンドボックス下で `mkdir` は使わず、`Write` による暗黙の親ディレクトリ作成に頼る）
2. 作成されたファイル一覧を `ls -la` で提示
3. 使い方を案内:

```
✅ Skill「<name>」を作成しました。

📁 場所: .claude/skills/<name>/
📄 ファイル:
  - SKILL.md
  - ...

▶ 使い方:
  - 次回以降の会話で「<起動フレーズ例1>」などと言えば呼び出されます
  - 明示的に起動したい場合は /<name> （スラッシュコマンドとして使える場合）
  - Skill のリロードは /reload もしくはセッション再起動が必要な場合あり

🔄 次のステップ:
  - 別会話でトリガーワードを試して動作確認
  - 必要なら .claude/skills/<name>/SKILL.md を直接編集して調整
  - Git にコミット: `git add .claude/skills/<name>/ && git commit -m "add skill: <name>"`
```

# 失敗・エッジケース

## サンドボックスで書き込み不可
`Write` ツールが拒否される場合:
- `.claude/settings.local.json` の `sandbox.filesystem.allowWrite` に `./.claude/skills/**` を追加
- 加えられない場合は managed settings 側で禁止されている。ユーザに依頼するか、一時ディレクトリ (`$TMPDIR`) に生成してから手動で移動してもらう

## 名前衝突
既存 Skill と同名の場合:
- Phase 4 で別名を再提案
- 上書き更新したい場合は本 Skill の範囲外 → 「既存ファイルを直接編集しましょう」と案内

## 会話コンテキストが薄い
「何をしたか」の情報が不足している場合:
- Phase 3 で徹底的にヒアリング
- それでも情報が足りなければ「まだ Skill 化には情報が足りないので、一度通しで作業してから改めて呼んでください」と案内

## Skill化に向かない作業
以下は Skill 化せず別手段を案内:
- 一度きりのタスク → CLAUDE.md へのメモか `/memory` へ
- プロジェクト固有情報 → CLAUDE.md や memory
- 秘密情報を含む手順 → 絶対に Skill に含めない

# 実行時の心構え

- **盲目的に Skill 化しない**: 会話の中には「失敗した試行錯誤」も含まれる。最終的にうまくいった手順だけを抽出する
- **抽象化しすぎない**: 具体的なコマンド・パスを残したほうが再利用時に迷わない
- **短く保つ**: SKILL.md は 200 行以内を目安。長大になりそうなら `reference/` に切り出す
- **description が最重要**: 将来の Claude が「この作業は Skill 化されているか」を判断する唯一の手がかり。具体的なトリガーフレーズを必ず含める
