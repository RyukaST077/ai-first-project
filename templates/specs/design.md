# 設計書テンプレート

---
**目的**: 実装者が異なっても一貫した実装ができるよう十分な詳細を提供し、解釈のブレを防止する。

**アプローチ**:
- 実装判断に直接影響する必須セクションを含める
- 実装ミスの防止に不可欠でない限り、任意セクションは省略する
- 機能の複雑さに応じて詳細度を調整する
- 長文より図表を優先する

**警告**: 1000行に近づく場合は機能が複雑すぎる兆候であり、設計の簡素化を検討すべきです。
---

> 読みやすさが向上する場合はセクションの並びを変更しても構いません（例: 要件トレーサビリティを先に出す、データモデルをアーキテクチャに近づける等）。各セクション内は **概要 → スコープ → 判断 → 影響/リスク** の流れを保ち、レビュー時に一貫してスキャンできるようにしてください。

## システムフロー

非自明なフローの説明に必要な図のみを記載します。純粋な Mermaid 構文を使用してください。代表的なパターン:
- シーケンス図（複数関係者のやり取り）
- プロセス / 状態遷移図（分岐ロジックやライフサイクル）
- データ / イベントフロー図（パイプライン、非同期メッセージング）

単純な CRUD 変更ではこのセクションをまるごと省略してかまいません。
> フローレベルの判断（ゲート条件、リトライ等）は図の後に簡潔に記載し、各ステップを文章で繰り返さないようにしてください。

## コンポーネントとインターフェース

コンポーネント単位の詳細に入る前に、クイックリファレンスを提供します。

- サマリは表または簡潔なリストで表現できます。表の例:
  | コンポーネント | ドメイン/レイヤー | 目的 | 要件カバレッジ | 主要依存 (P0/P1) | 契約 |
  |----------------|-------------------|------|----------------|------------------|------|
  | ExampleComponent | UI | XYZ を表示 | 1, 2 | GameProvider (P0), MapPanel (P1) | Service, State |
- 新たな境界（ロジックフック、外部統合、永続化など）を導入するコンポーネントのみ、詳細ブロックを必須とします。単純な表示用コンポーネントはサマリ行と短い実装メモで代替可能です。

詳細ブロックはドメインまたはアーキテクチャレイヤーでグループ化します。各詳細コンポーネントでは要件IDを `2.1, 2.3` のように列挙します（「Requirement」は付けない）。複数のUIコンポーネントが同じ契約を共有する場合は、コードブロックを重複させず、基底インターフェースやProps定義を参照してください。

## エラー処理

### エラー戦略
各エラー種別に対する具体的なエラー処理パターンとリカバリ機構。

### エラーカテゴリと応答
**ユーザーエラー**（4xx）: 不正な入力 → フィールド単位のバリデーション / 未認可 → 認証ガイド / 見つからない → ナビゲーション支援
**システムエラー**（5xx）: インフラ障害 → 段階的縮退 / タイムアウト → サーキットブレーカー / リソース枯渇 → レートリミット
**ビジネスロジックエラー**（422）: ルール違反 → 条件の説明 / 状態競合 → 遷移ガイド

**プロセスフローの可視化**（複雑なビジネスロジックがある場合）:
ビジネスワークフローを伴う複雑なエラーシナリオの場合のみ Mermaid フローチャートを含める。

### モニタリング
エラー追跡、ロギング、ヘルスモニタリングの実装。

- ユニットテスト: コア関数・モジュールから3〜5項目（例: 認証メソッド、サブスクリプションロジック）
- 結合テスト: コンポーネント横断フロー3〜5項目（例: Webhook処理、通知）
- E2E/UIテスト（該当する場合）: 重要なユーザーパス3〜5項目（例: フォーム、ダッシュボード）
- 性能/負荷テスト（該当する場合）: 3〜4項目（例: 並行処理、大量処理）
