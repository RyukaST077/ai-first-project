# API 標準

[目的: 命名、構造、認証、バージョニング、エラーに関する一貫した API パターン]

## 思想
- 予測可能でリソース指向の設計を優先
- 契約は明示的にし、破壊的変更を最小化
- セキュアバイデフォルト（認証優先、最小権限）

## エンドポイントパターン
```
/{version}/{resource}[/{id}][/{sub-resource}]
```
例:
- `/api/v1/users`
- `/api/v1/users/:id`
- `/api/v1/users/:id/posts`

HTTP メソッド:
- GET（取得、安全、冪等）
- POST（作成）
- PUT/PATCH（更新）
- DELETE（削除、冪等）

## リクエスト/レスポンス

リクエスト（典型例）:
```json
{ "data": { ... }, "metadata": { "requestId": "..." } }
```

成功:
```json
{ "data": { ... }, "meta": { "timestamp": "...", "version": "..." } }
```

エラー:
```json
{ "error": { "code": "ERROR_CODE", "message": "...", "field": "optional" } }
```
（ルールは error-handling を参照）

## ステータスコード（パターン）
- 2xx: 成功（200 取得、201 作成、204 削除）
- 4xx: クライアント起因（400 バリデーション、401/403 認証・認可、404 未存在）
- 5xx: サーバー起因（500 汎用、503 利用不可）
結果を最もよく表すステータスを選択すること。

## 認証
- 認証情報は標準的な位置に配置
```
Authorization: Bearer {token}
```
- 未認証のリクエストはビジネスロジックの前に拒否

## バージョニング
- URL / ヘッダー / メディアタイプのいずれかでバージョン管理
- 破壊的変更 → 新バージョン
- 非破壊的変更 → 同一バージョン
- 廃止予定期間と告知を提供

## ページング/フィルタリング（該当する場合）
- ページング: `page`、`pageSize` またはカーソルベース
- フィルタリング: 明示的なクエリパラメータ
- ソート: `sort=field:asc|desc`
ページング情報は `meta` に含めて返す。

---
_エンドポイントカタログではなく、パターンと判断に焦点を当てる。_
