# テスト標準

[目的: 何をテストするか、テストをどこに置くか、どう構造化するかの指針]

## 思想
- 実装ではなく振る舞いをテスト
- 高速で信頼性の高いテストを優先、脆いモックを最小化
- クリティカルパスを深くカバー、100% より広さ重視ではない

## 構成
選択肢:
- 同一配置: `component.tsx` + `component.test.tsx`
- 分離: `/src/...` と `/tests/...`
どちらか1つをデフォルトに、理由付きで例外を許容。

命名:
- ファイル: `*.test.*` または `*.spec.*`
- スイート: テスト対象、ケース: 期待する振る舞い

## テスト種別
- Unit: 単一ユニット、依存はモック、非常に高速
- Integration: 複数ユニットの結合、外部のみモック
- E2E: 全体フロー、モックは最小限、重要なユーザージャーニーのみ

## 構造（AAA）
```typescript
it('does X when Y', () => {
  // Arrange
  const input = setup();
  // Act
  const result = act(input);
  // Assert
  expect(result).toEqual(expected);
});
```

## モックとデータ
- 外部（API/DB）をモック、テスト対象自体はモックしない
- ファクトリ/フィクスチャを使用、テスト間で状態をリセット
- テストデータは最小限で意図を明確に

## カバレッジ
- 目標: [全体の％]、クリティカル領域はより高く
- CI でしきい値を強制、例外はレビューで理由を確認
