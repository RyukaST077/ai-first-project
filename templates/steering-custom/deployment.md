# デプロイメント標準

[目的: 明確な環境定義とパイプラインパターンによる、安全で再現可能なリリース]

## 思想
- 自動化し、デプロイ前にテスト、デプロイ後に検証
- 高速ロールバック可能な段階的ロールアウトを優先
- 本番変更は観測可能かつ可逆であること

## 環境
- Dev: 高速イテレーション、デバッグ有効
- Staging: 本番相当、リリース検証
- Prod: ハードニング、モニタリング、最小権限

## CI/CD フロー
```
Code → Test → Build → Scan → Deploy (staged) → Verify
```
原則:
- テスト/スキャンでは早期失敗、デプロイをブロック
- アーティファクトビルドは再現可能（lock ファイル、バージョン固定）
- 本番は手動承認、監査可能な記録

## デプロイ戦略
- ローリング: インスタンスを段階的に置き換え
- ブルーグリーン: 2つのプール間でトラフィックを切替
- カナリア: 少数ユーザーから開始、健全性に応じて拡大
リスクプロファイルに応じて選択、デフォルトをドキュメント化。

## ゼロダウンタイムとマイグレーション
- ヘルスチェックでトラフィックを制御、グレースフルシャットダウン
- ロールアウト中は DB の後方互換変更
- マイグレーションは別ステップ、ロールバック経路をテスト

## ロールバック
- 直前バージョンを常にデプロイ可能状態に、切り戻しを自動化
- 修正前進より高速、発動条件をドキュメント化

## 設定とシークレット
- 12-factor に従い env で設定、シークレットはコミット禁止
- シークレットマネージャー、ローテーション、最小権限、アクセス監査
- 起動時に必要な環境変数を検証

## ヘルスとモニタリング
- エンドポイント: `/health`、`/health/live`、`/health/ready`
- レイテンシ、エラー率、スループット、飽和度を監視
- SLO 違反/スパイクでアラート、疲労回避のためチューニング

## インシデント対応と DR
- 標準プレイブック: 検知 → 評価 → 緩和 → 連絡 → 解決 → ポストモーテム
- 保持期間付きバックアップ、リストアをテスト、RPO/RTO を定義

---
_ロールアウトパターンとセーフガードに焦点を当てる。プロバイダ固有の手順は含めない。_
