# 認証・認可 標準

[目的: 認証モデル、トークン/セッションのライフサイクル、権限チェック、セキュリティの統一]

## 思想
- 明確な分離: 認証（誰か）と認可（何ができるか）
- セキュアバイデフォルト: 最小権限、フェイルクローズ、短命トークン
- UX を意識: リスクが高い箇所には摩擦を、それ以外はスムーズに

## 認証

### 方式（選定 + 根拠）
- 候補: JWT、セッション、OAuth2、ハイブリッド
- 選定: [採用する方式] 理由 [選定理由]

### フロー（高レベル）
```
1) ユーザーが身元を証明（認証情報またはプロバイダ）
2) サーバーが検証し、トークン/セッションを発行
3) クライアントはリクエストごとにトークンを送信
4) サーバーはトークンを検証して処理を続行
```

### トークン/セッションのライフサイクル
- 保存: httpOnly クッキーまたは Authorization ヘッダー
- 有効期限: アクセストークンは短命、リフレッシュはより長め（使用する場合）
- リフレッシュ: トークンをローテーション、失効を尊重
- 失効: ログアウト/侵害時にブラックリスト化またはローテーション

### セキュリティパターン
- TLS を強制、可能な限り JS にトークンを公開しない
- トークンを audience/issuer にバインドし、クレームは最小限
- 機密操作にはデバイスバインディングや IP/リスクチェックを検討

## 認可

### 権限モデル
- いずれかを採用: RBAC / ABAC / オーナーシップベース / ハイブリッド
- ロールや属性は中央で定義し、コードベース全体にハードコードしない

### チェック（強制する場所）
- ルート/ミドルウェア: 粗粒度のゲート
- ドメイン/サービス: 細粒度の判断
- UI: 条件付きレンダリング（セキュリティの拠り所にしない）

パターン例:
```typescript
requirePermission('resource:action'); // route
if (!user.can('resource:action')) throw ForbiddenError(); // domain
```

### オーナーシップ
- パターン: オーナー または 特権ロールが操作可能
- 変更前にエンティティ境界で検証

## パスワードと MFA
- パスワード: 強いポリシー、ハッシュ化（bcrypt/argon2）、平文保存禁止
- リセット: 時限・単回使用のトークン、ユーザーに通知
- MFA: リスクのある操作ではステップアップ（ポリシーに基づく）

## API 間認証
- API キーまたは OAuth クライアントクレデンシャルを使用
- キーのスコープは最小限、ローテーションし利用を監査
- アイデンティティ（ユーザー/キー）単位でレート制限
