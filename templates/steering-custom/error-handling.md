# エラー処理標準

[目的: エラーの分類、形式、伝播、ロギング、モニタリングの統一]

## 思想
- 可能な限りフェイルファスト、システム境界ではグレースフルに縮退
- スタック全体で一貫したエラー形式（人間にも機械にも読みやすい）
- 既知のエラーは発生源近くで処理、未知のエラーはグローバルハンドラへ

## 分類（発生源で処理方針を決定）
- クライアント: 入力/検証/ユーザー操作の問題 → 4xx
- サーバー: システム障害/予期しない例外 → 5xx
- ビジネス: ルール/状態違反 → 4xx（例: 409）
- 外部: サードパーティ/ネットワーク障害 → コンテキストに応じて 5xx または 4xx にマッピング

## エラーの形式（単一の正規フォーマット）
```json
{
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable message",
    "requestId": "trace-id",
    "timestamp": "ISO-8601"
  }
}
```
原則: 安定したコード enum、機密情報を含めない、トレース情報を含める。

## 伝播（変換する場所）
- API レイヤー: ドメインエラー → HTTP ステータス + 正規ボディ に変換
- サービスレイヤー: 型付きビジネスエラーをスロー、文字列ベースのエラーを避ける
- データ/外部レイヤー: プロバイダのエラーを安全で行動可能なコードでラップ
- 未知のエラー: グローバルハンドラへバブルアップ → 500 + 汎用メッセージ

パターン例:
```typescript
try { return await useCase(); }
catch (e) {
  if (e instanceof BusinessError) return respondMapped(e);
  logError(e); return respondInternal();
}
```

## ロギング（ノイズよりコンテキスト）
記録するもの: 操作、userId（利用可能なら）、code、message、stack、requestId、最小限のコンテキスト。
記録しないもの: パスワード、トークン、シークレット、完全な PII、機密を含むフルボディ。
レベル: ERROR（失敗）、WARN（回復可能/境界ケース）、INFO（主要イベント）、DEBUG（診断）。

## リトライ（安全な場合のみ）
リトライ可: ネットワーク/タイムアウト/一時的な 5xx かつ冪等な操作。
リトライ不可: 4xx、ビジネスエラー、非冪等なフロー。
戦略: 指数バックオフ + ジッター、試行回数に上限、冪等性キーを要求。

## モニタリングとヘルス
追跡: コード/カテゴリ別のエラー率、レイテンシ、飽和度、スパイク/SLI 違反でアラート。
ヘルス公開: `/health`（live）、`/health/ready`（ready）。エラーとトレースをリンク。
