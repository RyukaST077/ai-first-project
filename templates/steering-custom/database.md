# データベース標準

[目的: スキーマ設計、クエリ、マイグレーション、整合性を導くガイド]

## 思想
- まずドメインをモデル化し、正しさを確保してから最適化
- 明示的な制約を優先、不変条件は DB に強制させる
- 必要なものだけクエリ、最適化は計測してから

## 命名と型
- テーブル: `snake_case`、複数形（`users`、`order_items`）
- カラム: `snake_case`（`created_at`、`user_id`）
- FK: `{table}_id` が `{table}.id` を参照
- 型: タイムゾーン対応のタイムスタンプ、強い型の ID、金額は精度のある型

## リレーション
- 1:N: 子側に FK
- N:N: 複合キーの中間テーブル
- 1:1: FK + UNIQUE

## マイグレーション
- イミュータブルなマイグレーション、常にロールバックを用意
- 小さく焦点を絞ったステップ、本番以外で先にテスト
- 命名: `{seq}_{action}_{object}`（例: `002_add_email_index`）

## クエリパターン
- 単純な CRUD や安全性には ORM、複雑・性能重視には生 SQL
- N+1 を避ける（eager load / バッチ化）、大きな結果はページング
- FK や頻繁にフィルタ/ソートするカラムにはインデックス

## コネクションとトランザクション
- プーリングを使用（サイズ/タイムアウトはワークロードに応じて）
- 作業単位ごとに1接続、速やかに返却/クローズ
- 複数ステップの変更はトランザクションでラップ

## データ整合性
- NOT NULL / UNIQUE / CHECK / FK 制約を使用
- 適切な箇所では DB 側でも検証（多層防御）
- 一貫した派生値には生成列を優先

## バックアップとリカバリ
- 保持期間付きの定期バックアップ、リストアをテスト
- RPO/RTO 目標をドキュメント化し、バックアップジョブを監視

---
_パターンと判断に焦点を当てる。環境固有設定は含めない。_
