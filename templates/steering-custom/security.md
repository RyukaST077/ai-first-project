# セキュリティ標準

[目的: 入力検証、認可、シークレット、データ保護に関するパターンでセキュリティ姿勢を定義]

## 思想
- 多層防御、最小権限、セキュアバイデフォルト、フェイルクローズ
- 境界で検証、用途に応じてサニタイズ、入力を決して信用しない
- 認証（誰か）と認可（何ができるか）を分離

## 入出力
- API 境界と UI フォームで検証、型と制約を強制
- 送信先に応じてサニタイズ/エスケープ（HTML、SQL、シェル、ログ）
- ブロックリストよりも許可リストを優先、最小限の詳細で早期に拒否

## 認証と認可
- 認証: アイデンティティを検証、短命トークン/セッションを発行
- 認可: 操作前に権限チェック、デフォルトで拒否
- ポリシーを集約、チェックをコード全体で重複させない

パターン:
```typescript
if (!user.hasPermission('resource:action')) throw ForbiddenError();
```

## シークレットと設定
- シークレットはコミットしない、シークレットマネージャーまたは env に保存
- 定期ローテーション、アクセス監査、最小スコープ
- 起動時に必要な環境変数を検証、不足時はフェイルファスト

## 機密データ
- 収集を最小化、ログではマスキング/リダクション、保存時/転送時に暗号化
- ロール/必要性に応じてアクセスを制限、機密レコードへのアクセスを追跡

## セッション/トークンのセキュリティ
- 可能な限り httpOnly + secure クッキー、全面的に TLS
- 短い有効期限、リフレッシュ時にローテーション、ログアウト/侵害時に失効
- トークンを audience/issuer にバインド、クレームは最小限

## ロギング（セキュリティ意識）
- 認証試行、権限拒否、機密操作をログに記録
- パスワード、トークン、シークレット、完全な PII はログに残さない、フルボディも避ける
- イベント相関のため requestId とコンテキストを含める

## ヘッダーとトランスポート
- TLS を強制、HSTS
- セキュリティヘッダーを設定（CSP、X-Frame-Options、X-Content-Type-Options）
- モダンな暗号を優先、脆弱なプロトコル/暗号を無効化

## 脆弱性姿勢
- セキュアなライブラリを優先、依存関係を最新に保つ
- CI で静的/動的スキャン、追跡して修復
- 一般的な脆弱性クラスをチームに教育、上記パターンにコード化
